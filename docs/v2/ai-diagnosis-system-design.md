# K12consult 香港转学 AI 诊断系统：全局设计总纲 (V2.0)

## 1. 项目愿景与定位

### 1.1 项目核心
K12consult 不是单点问答工具，而是“规则引擎 + 学校数据 + 服务流程”一体化系统。目标是把复杂转学决策转成可执行结论，并通过会员化服务持续提升转化与复购。

### 1.2 核心价值主张
- **卖决策而非知识**：基于身份、时间窗、能力、预算做硬筛选。
- **动态事实驱动**：学校数据快照保障判断可追溯。
- **专家服务闭环**：AI 初诊到管理员分配，再到顾问跟进闭环。

---

## 2. 角色与权限边界

### 2.1 平台角色
1. **访客 (Guest)**：未登录，允许进入问卷流程。
2. **家长会员 (Parent Member)**：可查看报告、管理孩子档案、提交咨询请求。
3. **顾问 (Consultant)**：处理被分配案例，持续跟进并完成成交或转资源库。
4. **管理员 (Admin)**：筛选线索、补资料、分派顾问、监控 SLA。

### 2.2 角色映射规范
- 文档角色：`parent_user / consultant / admin / super_admin`
- 运行时兼容：`user / consultant / admin`
- 原则：角色命名可兼容，但权限判定必须走统一 RBAC 规则，不走页面硬编码。

### 2.3 权限最小化原则
- 用户仅可访问本人 case。
- 顾问仅可访问已分配 case。
- 管理员可访问运营范围内全部 case。
- 内部标签（意向等级、内部备注、派单策略）用户端不可见。

---

## 3. 端到端主流程 (V2.0)

1. **获客进入**：访客进入问卷，记录渠道与会话。
2. **问卷提交**：提交后生成诊断任务与报告草稿。
3. **会员绑定**：通过登录/注册绑定联系方式与历史 case。
4. **报告查看**：家长会员查看结论并提交咨询意向。
5. **管理员筛选**：补资料、判定去向（转顾问/培育/关闭）。
6. **顾问执行**：接收派单、跟进处理、成交关闭或未成交转资源库。

## 4. 用户登录与会员管理优化 (本次更新核心)

### 4.1 设计目标
- 降低首次进入门槛（先体验再绑定）。
- 保证账户唯一性（同一联系方式不产生多账号孤岛）。
- 用会员状态驱动服务动作（而非仅靠页面跳转）。
- 让“登录、报告、咨询、跟进”形成连续体验。

### 4.2 登录策略定稿 (A 方案)
1. **问卷前不强制登录**：访客可直接填写问卷。
2. **关键动作强登录**：提交咨询意向、补资料、查看“我的案例”前要求登录。
3. **统一账号入口**：家长、顾问、管理员共用同一登录入口与账户体系。
4. **主登录方式**：手机号验证码登录（Primary）。
5. **备用登录方式**：邮箱 + 密码登录（Secondary）。
6. **会话策略**：`httpOnly + sameSite=lax`，默认 7 天会话，敏感操作要求二次校验。

### 4.3 登录交互模式 (参考 Google)
1. 第一步：只输入账号标识（手机号或邮箱）。
2. 第二步：根据账号状态进入验证动作：
   - 已有账号：手机号走验证码；邮箱走密码（可选验证码二次验证）。
   - 新账号：手机号验证码注册，补全基础资料后完成绑定。
3. 登录成功后按角色跳转目标工作区，并保留 `next` 回跳能力。

### 4.4 会员生命周期状态机
`guest -> identified -> registered -> active_member -> service_in_progress -> dormant -> reactivated`

状态说明：
- `guest`：匿名访问，仅有临时会话。
- `identified`：已留联系方式，尚未完成会员绑定。
- `registered`：已完成账号注册并绑定历史线索。
- `active_member`：有可访问报告或待处理 case。
- `service_in_progress`：已进入顾问服务阶段。
- `dormant`：长期无活跃行为，进入培育池。
- `reactivated`：被运营唤回并重新进入活跃状态。

### 4.5 会员权益分层 (第一版可执行)
1. **访客**：可填问卷、可看处理中状态，不可访问完整历史。
2. **注册会员**：可看本人报告、提交咨询意向、管理孩子档案。
3. **服务中会员**：可补资料、看咨询进展、查看顾问结论与会后动作。

> 第一版不引入复杂付费套餐，先按“服务阶段权益”分层，避免早期策略过重。

### 4.6 账号去重与合并规则
1. 以 `mobile / wechat_id / email` 任一命中做候选匹配。
2. 命中多条时按“最近活跃 + 已验证联系方式”优先归并。
3. 合并后保留所有 case 历史，不覆盖顾问与管理员操作记录。
4. 合并行为必须写审计日志（操作人、时间、影响对象）。

### 4.7 会员中心最小功能集
- 我的案例列表（按状态分组）
- 报告与咨询进度页（时间线）
- 联系方式与通知偏好管理
- 登录设备与安全设置（会话管理）

## 5. 核心功能设计

### 5.1 诊断与报告
- 输入：结构化问卷 + 标准化字段。
- 判定：规则引擎输出路径、风险、建议动作。
- 输出：AI 报告草稿 + 可回溯快照。

### 5.2 管理员工作台
- 线索评分、SLA 监控、去向决策。
- 管理员跟进记录与交接摘要结构化沉淀。

### 5.3 顾问工作台
- 交接包视图、跟进记录结构化录入、状态闭环追踪。
- 关键门禁：未填成交结论不可关闭为成交；未填未成交原因不可转资源库。

---

## 6. 数据与架构设计

### 6.1 账户与会员域
- `users`：统一身份主体（角色、状态、基础认证信息）。
- `user_identities`：登录标识绑定（mobile/email/wechat），支持一人多标识。
- `profiles`：角色扩展资料。
- `user_sessions`：统一会话追踪与设备管理。
- `login_challenges`：验证码/二次验证挑战记录（含过期与校验状态）。
- `membership_state`（可并入 profile/case 视图）：记录会员生命周期状态与更新时间。

### 6.2 业务主线
- `users -> students -> cases -> questionnaire_responses -> diagnostic_jobs -> reports`
- 管理与服务层：`admin_follow_up_records -> consultant_assignments -> case_follow_ups`

### 6.3 快照与审计
- 诊断快照：规则版本、模型版本、学校数据版本。
- 操作审计：登录、角色变更、账号合并、状态流转、关闭动作。

---

## 7. 核心判定逻辑

1. 身份准入矩阵
2. 时间窗口矩阵
3. 英文风险公式
4. 服务门禁规则（咨询完成门禁、关闭门禁、补资料门禁）

---

## 8. 运营与服务标准

1. 核心线索 1 小时首次响应。
2. 补资料窗口默认 7 天，超期进入培育策略。
3. 登录与会员转化指标纳入运营看板：
   - 问卷提交 -> 注册转化率
   - 注册 -> 咨询意向转化率
   - 咨询意向 -> 顾问接单转化率

---

## 9. 安全与异常处理

- 用户数据隔离与最小权限访问。
- 登录失败与异常行为风控（频控、黑名单、异常设备提醒）。
- 诊断失败自动重试 + 人工兜底。
- Supabase/本地存储异常时保底一致性策略（fallback + 审计日志）。

---

## 10. 路线图 (登录与会员专项)

1. **Phase 1（当前）**：统一会话、角色权限、主流程门禁。
2. **Phase 2**：会员中心（我的案例、进度、资料管理）。
3. **Phase 3**：账号自动归并与精细化培育策略。
4. **Phase 4**：会员分层运营实验（不同服务包转化测试）。
