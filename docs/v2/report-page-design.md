# K12consult V2 报告页设计（严格版）

基准文档：`docs/v2/diagnosis-report.md`（模板 V1.2）  
目标页面：`/result/[submissionId]`

---

## 1. 设计目标

报告页只做一件事：**完整交付一份有判断力、可行动、可转化的诊断报告**。  

本设计要求严格遵循 `diagnosis-report.md` 的章节结构，不允许在核心报告区随意增删或打乱顺序。

---

## 2. 强约束（必须遵守）

1. 报告正文固定 **0-7 共 8 个章节**，顺序与标题保持一致。
2. 每个章节必须有对应数据，不可用“空模块”占位。
3. 第 6 章必须包含动作按钮：
   - `立即预约资深顾问`（单独一行醒目展示）
4. 所有用户可见文案中文优先；可保留模板中的英文括注（如 `The Verdict`）。
5. 报告核心区不混入运营后台动作（补资料、状态流转等）。

---

## 3. 页面信息架构

从上到下：

1. 顶部导航：`返回仪表板`、`重新填写`（弱化）
2. 报告头部（基础信息 + 结论摘要）
3. 正文第 0-7 章（严格按模板）
4. 报告后操作区（非正文）：
   - 咨询意向提交
   - 补资料入口（仅状态允许时显示）

说明：第 4 部分属于“服务动作区”，不计入报告正文章节。

---

## 4. 章节级设计（严格映射模板）

## 4.1 第 0 章：基本信息
- 展示字段：
  - 学生主体
  - 当前年级
  - 身份状态
  - 预算匹配
- 数据优先级：
  - `lead.questionnaireResponse.responseJson`
  - `lead.currentReport.summary`
  - `lead.result`（兜底）

## 4.2 第 1 章：核心结论 (The Verdict)
- 展示内容：
  - 一句话核心结论
  - `AI 专家判定` 引导块（身份预警 + 总体风险摘要）
- 不允许：
  - 泛化废话
  - 与规则结果冲突的表述

## 4.3 第 2 章：核心诊断维度
- 2.1 申请时机与窗口：
  - 判定状态
  - 详细评估
- 2.2 语言衔接风险：
  - 预警等级
  - 用户当前 vs 目标池门槛
  - 专家建议

## 4.4 第 3 章：目标学校穿透建议
- 逐校展示：
  - 学校名
  - 适配度评估（`/10`）
  - 关键卡点
  - 顾问私货
  - 学费提示
- 推荐数量：`3` 所（默认），若不足按实际返回。

## 4.5 第 4 章：深度风险预警
- 使用列表渲染风险条目：
  - 标题
  - 风险内容
- 若无预警：显示单条“当前无硬性阻断，可继续推进”。

## 4.6 第 5 章：后续行动清单
- 分两阶段固定结构：
  - 第一阶段：下一步 24 小时（2 条）
  - 第二阶段：1-2 周（2 条）
- 不允许出现“建议考虑”类空泛动作。

## 4.7 第 6 章：专家人工介入引导
- 固定引导句：
  - `由于你的案例涉及 {复杂标签}，建议立即预约资深顾问进行深度评估。`
- 固定按钮：
  - 按钮：`立即预约资深顾问`（提交/更新咨询意向）
  - 视觉：不放并列次按钮，按钮单独一行、满宽高强调


## 4.8 第 7 章：诊断依据声明
- 固定显示：
  - 规则版本
  - 模型引擎
  - 学校库快照日期
- 数据来源：
  - `lead.diagnosticJob.versionSnapshot`
  - `lead.diagnosticResult.schoolDataSnapshotJson`

---

## 5. 数据契约（报告页最小必需）

为保证“严格模板渲染”，建议报告接口输出结构化对象：

```json
{
  "reportView": {
    "basicInfo": {},
    "verdict": {},
    "diagnosticDimensions": {},
    "recommendedSchools": [],
    "criticalWarnings": [],
    "nextSteps": {
      "phaseImmediate": [],
      "phasePreparation": []
    },
    "consultationGuide": {},
    "meta": {}
  }
}
```

当前阶段可由 `lead.currentReport.summary + lead.result + diagnosticResult` 组装，后续沉淀到 `reports.summary_json`。

---

## 6. 视觉与排版规范

1. 风格与主页一致：极简、留白、低装饰。
2. 报告正文使用单列阅读流，宽度建议 `720-820px`。
3. 章节标题层级固定：`H2(章) / H3(小节)`。
4. 风险与行动用列表，不做卡片拼贴。
5. 第 6 章按钮是页面最高对比元素，不放“电话拨打”并列入口。

---

## 7. 当前页面改造要求（对 `app/result/[submissionId]/page.js`）

1. 将“路径初判、四维评分、主要风险点”等旧模块并入模板对应章节，避免重复表达。
2. 报告正文按 0-7 章重排后再渲染服务动作区。
3. `咨询意向` 与 `补资料` 仅放在报告正文之后，不插入正文中段。
4. 保留权限校验与状态校验逻辑，不改变现有安全边界。

---

## 8. 验收标准（UAT）

1. 报告页首屏可见“第 0 章基本信息 + 第 1 章核心结论”。
2. 页面存在且仅存在一次 0-7 章节完整结构。
3. 第 6 章仅保留预约顾问按钮，且可成功提交咨询意向。
4. 第 7 章能看到版本与快照日期，不为空。
5. 无论数据是否完整，页面不出现空白区块或报错。

---

## 9. 实现文件映射

- 页面：`app/result/[submissionId]/page.js`
- 报告生成与摘要：`lib/intake.js`
- 结果接口：`app/api/results/[submissionId]/route.js`
- 诊断模板基准：`docs/v2/diagnosis-report.md`
