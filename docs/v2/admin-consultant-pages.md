# 香港转学 AI 诊断系统：管理员跟进页与顾问工作台结构设计

## 1. 结论

为了配合“三段式咨询承接流程”，我们需要设计两个核心工作空间：

1. **管理员跟进页**：核心目标是 **“由线索变确定性案例”**
2. **顾问工作台**：核心目标是 **“由案例变交付结论”**

管理员页重在 **“填空和判断”**，顾问页重在 **“阅读和输出”**。

---

## 2. 管理员跟进页结构设计

这是管理员联系用户、补齐信息并决定去向的主战场。

### 视图 A：跟进池列表
- **核心列**：Case 编号、用户姓名、孩子年级、提交时间、意向状态、SLA 倒计时
- **操作**：认领、分配、直接联系（拨号/复制微信）

### 视图 B：案例跟进详页（重点）
建议采用三栏式结构：

#### 左栏：背景资料区
- 用户原始问卷答案
- AI 初诊报告摘要
- 之前的顾问备注（如有）

#### 中栏：跟进操作区（核心表单）
管理员在联系用户时边听边填：
1. **咨询意愿确认**：高/中/低
2. **目标时间轴**：这学期/9月/明年
3. **学费预估接受度**：低/中/高
4. **本次咨询重点**（多选）：选校、路径、语言、手续、综合等
5. **资料补齐状态**：标记是否已上传成绩单、英文成绩等

#### 右栏：处理结论区
- **操作按钮组**：
  - [可转顾问] -> 填写交接摘要、选择顾问、确认时间
  - [需补资料] -> 标记缺失项，状态转入 `awaiting_user_info`
  - [暂不转顾问] -> 选原因入 `nurturing` 池
  - [关闭案例] -> 标记原因关闭

---

## 3. 顾问工作台结构设计

这是顾问执行管理员派单的轻量工作台，聚焦“接单 -> 跟进 -> 关闭”。

### 视图 A：待处理案例看板
- **核心列**：孩子姓名、当前状态、管理员交接摘要、最近更新时间、是否临近 SLA
- **核心动作**：打开案例、更新跟进备注、推进状态

### 视图 B：案例跟进详情页（重点）
设计原则：**最少字段、最快推进、清晰留痕**。

#### 1. 管理员交接包（置顶）
- 意愿等级、预算水平、推进时间
- 交接摘要
- 管理员特别提醒

#### 2. 跟进执行区（顾问主操作）
- 跟进备注（每次沟通记录）
- 下一步动作（待办说明）
- 下次联系时间（可选）
- 状态推进按钮：
  - `consult_assigned`（已接单）
  - `follow_up`（跟进中）
  - `closed`（已成交关闭）
  - `nurturing`（未成交转资源库）

#### 3. 关闭校验（避免提前关单）
- 成交关闭（`closed`）前必须填写成交结论
- 未成交转资源库（`nurturing`）前必须填写未成交原因与后续建议
- 关闭或转资源库后仅管理员可重新激活

## 4. 已确认业务决策落实

### 4.1 管理员跟进 SLA 监控
- 在管理员列表页及详情页置顶展示 **“首次联系倒计时”**
- 超过 2 小时红色预警，超过 24 小时进入异常名单

### 4.2 Nurturing 池管理逻辑
- 这是一个独立的二级列表。
- **操作**：
  - [重新激活] -> 用户主动联系或时机成熟后，管理员重新发起跟进。
  - [系统提醒日志] -> 显示自动化触达记录。

### 4.4 资源库管理逻辑（顾问后链路）
- 这是顾问跟进后“未成交”案例的归档池。
- **触发**：顾问在 `follow_up` 阶段判断当前无法成交。
- **操作**：
  - [转资源库] -> 状态改为 `nurturing`，填写未成交原因。
  - [重新激活] -> 管理员评估后重新进入 `admin_following` 或 `consult_assigned`。

### 4.3 待补资料管理机制
- 列表页显示 **“已等待补资料 X 天”**
- 达到 7 天时系统自动发出警告，管理员手动决定转入 nurturing。

## 5. 关键操作的字段级闭环

### 管理员转顾问瞬间的系统动作：
- 写入 `admin_follow_up_records`
- 写入 `consultant_assignments`
- 写入 `assigned_at`（接单时间）
- `case.status` = `consult_assigned`
- 顾问未成交时写入 `nurturing_reason` 并更新 `case.status = nurturing`

## 6. 需要你拍板的 1 个点

这个点会决定顾问执行规范力度。

### 决策点：顾问关单前是否要求“最少跟进记录数”？
#### 方案 A：不要求最少条数，只要有关闭原因即可
#### 方案 B：至少 1 条跟进记录
#### 方案 C：至少 2 条跟进记录（高规范）

**我的建议：选 B。**
第一版顾问系统以执行效率优先，避免“0 跟进直接关单”。

## 8. 页面路由拆分（2026-04-18）

为保证“管理员与顾问工作台彻底分开”，前端路由按职责拆分为两条独立主线：

### 8.1 管理员路由
- `/admin/workbench`：管理员主工作台（跟进池 + 状态看板）
- `/admin/cases/[id]`：管理员案例详情（跟进与交接）
- `/admin/nurturing`：培育池（二级列表）
- `/admin/sla`：SLA 异常清单

### 8.2 顾问路由
- `/advisor/workbench`：顾问主工作台（已派单与跟进中）
- `/advisor/cases/[id]`：顾问案例详情（接单、跟进、关闭）
- `/advisor/follow-up`：跟进清单（可选二级页）

### 8.3 兼容策略
- 旧入口保留兼容跳转：
  - `/admin` -> `/admin/workbench`
  - `/admin/leads/[id]` -> `/admin/cases/[id]`
  - `/advisor` -> `/advisor/workbench`
  - `/advisor/leads/[id]` -> `/advisor/cases/[id]`

### 8.4 接口拆分（已完成）
- 管理员案例接口：`/api/admin/cases/[id]`
  - 仅 `admin / super_admin` 可读写
- 顾问案例接口：`/api/advisor/cases/[id]`
  - `consultant / admin / super_admin` 可读
  - 顾问仅可修改顾问侧字段，不可改派、不可改管理员字段

### 8.5 二级页面能力填充（已完成）
- 管理员：
  - `/admin/nurturing` 已接入真实数据，支持查看培育时长、最近备注、快速回到案例并重新激活
  - `/admin/sla` 已接入两类异常清单：首次联系超时、待补资料超 7 天
- 顾问：
  - `/advisor/follow-up` 已接入跟进任务清单，聚焦“跟进中（follow_up）”与“待启动跟进（consult_assigned）”
  - `/advisor/schedule` 统一使用 `consult_assigned / follow_up` 两类顾问态进行排期展示
  - 顾问推进到 `follow_up` 前需存在至少 1 条顾问跟进记录（系统自动日志不计入）

说明：当前已完成 Phase A 路由拆分 + Phase B 案例接口拆分 + Phase C 二级页面能力填充；顾问端按“接单-跟进-成交关闭/未成交转资源库”执行。

## 9. 管理员管理系统升级方案（2026-04-18）

根据最新业务决策，管理员系统从“单一跟进台”升级为三大模块并行：

1. 管理顾问：新增、编辑、停用、负载管理
2. Case 总览与指派：查看所有 Case，支持手动/推荐/批量指派
3. Case 状态管理：按 V2 状态机推进，并在详情页沉淀操作记录

### 9.1 管理员首页结构

- `/admin/workbench` 增加三张模块入口卡：
  - `管理顾问`
  - `Case 总览与指派`
  - `Case 状态管理`
- 状态看板改为紧凑模式：`下拉选择状态 + 快捷状态标签`，按选择展示对应案例数与列表

### 9.2 新增页面边界

- `/admin/consultants`：顾问管理列表
- `/admin/consultants/new`：新增顾问
- `/admin/consultants/[consultantId]`：顾问编辑与履历
- `/admin/cases`：全量 Case 列表与指派中心
- `/admin/cases/[leadId]`：Case 详情与状态推进（既有页面继续承接）

### 9.3 关键规则补充

- 顾问删除采用软删除（`inactive`），不做硬删除
- 指派前后必须写入审计日志
- 状态推进保持字段约束（如排期时间、关闭原因、咨询结论摘要）
- `awaiting_user_info` 超 7 天进入 SLA 异常池，由管理员决策下一步
